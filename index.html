Use Case 01 · Rapid System Architect

At some point, you inherit books that no longer tie.

During periods of transition, context often lives across systems, spreadsheets, Slack threads, and individual workflows. The close still has to move forward, but the first challenge is making the operating picture visible enough to trust.

Sound familiar?
▦

Context is distributed

Important knowledge often lives in multiple places, not one clean source.

≠

History is hard to reconstruct

Balances may be correctable, but the path to get there is not always obvious.

≣

Methods evolve over time

Different people may have solved problems in different ways as the business changed.

△

Changes need care

Before adjusting prior work, Finance needs a clear trail from source activity to close impact.

This use case is the playbook.
Anchor to the last clean state, reconstruct what changed, and rebuild toward a close that is fast, accurate, and auditable.

A subsidiary had drifted out of sync, and the close still had to happen.

The AI workflow

streaming files into context…
1
Built one big working context
Pulled every trust-account statement, the ERP ledger exports, and six recordings of the original close into a single Claude context, so it could reason over the whole entity at once, not file by file.
2
Anchored to the last clean month
Had Claude reconstruct how the books were supposed to work (account structure, entry patterns, the funding cycle) from the verified baseline and the walkthroughs.
3
Interrogated what looked wrong
A long back-and-forth against the source data: "this balance is negative, why?", "trace this clearing entry to the bank," "these liabilities don't split out." Each question I asked surfaced a concrete problem:
Intercompany clearing booked in USD, not GBP
Liabilities collapsed into one catch-all
Payments made but never journaled
Parent funding receipts never recorded
A reclass that unbalanced the one clean month
A recurring accrual silently stopped
4
Verified, corrected, documented
Pinned each finding to a specific bank line, structured one cumulative self-balancing correction, and reviewed every entry before posting, with an audit narrative explaining the why.

What this proves

The hard part wasn't correcting the numbers; it was reconstructing why they drifted. Tracing each account from the last clean baseline back to source activity turned a potential multi-period reopening into one cumulative correction, delivered as an audit-ready package explaining the why behind every entry.

Use Case 02 · Knowledge Pipeline Engineer

The workflows are already there. The documentation isn't.

Record-to-report: the workflow behind the numbers spans people, systems, approvals, reconciliations, and exceptions. The goal is to turn that context into a visible process map the team can improve, control, and eventually automate.

So I turn every process conversation into a map
◉

1 · Conversation

Sit with each process owner and record the walkthrough. Capture it once, properly.

▤

2 · Transcript

Speech-to-text turns the call into a clean, searchable record of how it really works.

⊞

3 · Visual map

Feed the transcript to AI and get the process back as a diagram anyone can read.

⚑

4 · Pain points

The map surfaces the gaps and dependencies that quietly impact the downstream close.

One method, every process.
Run the same loop across order-to-cash, procure-to-pay, and hire-to-pay, and you've mapped how the company actually operates: reusable knowledge for the next close, the next hire, the next automation.

Where the information is derived, and what breaks downstream
1
Validate before billing
The CRM is the source of truth: dates, amounts, and contract terms are checked before setup. One bad input can throw off invoices and revenue from day one.
2
Terms drive recognition
Multi-year terms, amendments, and credits determine the revenue schedule. Miss one, and revenue lands in the wrong period or deferred revenue stops rolling cleanly.
3
Close exposes the gaps
Month-end reconciles billing to the ledger. Any variance usually traces back to handoff, amendments, or setup, so tighter upstream controls mean a faster, cleaner close.

What this proves

One conversation, and the whole cycle is legible: where every number originates, and which handoffs put downstream accuracy at risk. The same read maps any process the company runs.
Use Case 03 · Agent Workflow Architect

Learning the file is step one.
Making it run without me is the goal.

So every workbook I take on moves through two phases
Phase 1doing now
File Intelligence
▸Feed the workbook to AI and walk through how it works and updates, step by step
▸Reverse-engineer the underlying logic: the inputs, the rules, the outputs
▸Rebuild the static spreadsheet to be dynamic and scalable, with logic that flexes with new data instead of breaking
▸Write the Agent Tab, a plain-language operating guide written for an agent
→
Phase 2next
Agent Deployment
▸Deploy an agent on the workflow. It operates the file from the Agent Tab
▸You feed it the data; it builds and updates the schedule
▸You stay in the loop and review every entry before it posts
Build the instructions once, deploy when ready.
Each Agent Tab is a skill definition that outlasts me. The workbook becomes a workflow an agent can run. I'm starting with the lowest-risk, highest-repeat close areas.

First I make the file agent-ready. Then I build the agent.
